# AGENTS.md

## 项目目标
wechat-support-agent 是一个“微信远程技术支持 Agent 分身”的 MVP 项目。第一版不接真实微信，只通过 Web 页面模拟客户在微信中咨询技术问题。Agent 需要完成问题接待、信息收集、问题分类、排查建议和工单摘要生成。

## 技术栈
- 后端：Python FastAPI
- 前端：HTML + CSS + JavaScript
- 数据库：SQLite
- 知识库：Markdown
- 第一版不接真实微信
- 第一版不接真实 LLM API，先使用 mock
- 后续可扩展企业微信或微信客服接口

## 多 Agent 分工
项目采用 Main Agent 监控 + 子 Agent 分工的方式开发。Main Agent 负责阶段边界、任务拆分、代码审查和最终集成；子 Agent 按职责在限定目录内完成小步变更。

## Main Agent 职责
- 维护项目阶段计划、验收标准和任务边界。
- 拆分 Backend、Frontend、Knowledge、Test、Docs 等子任务。
- 审查子 Agent 产出，确保没有越权修改和提前实现。
- 控制提交粒度，避免一次性大改。
- 负责最终集成、状态汇总和下一阶段建议。

## Backend Agent 职责
- 负责 FastAPI 后端相关设计和实现。
- 负责 SQLite 数据模型、mock Agent 服务和工单摘要接口的后续实现。
- 当前第 0 阶段不得实现任何业务接口。

## Frontend Agent 职责
- 负责 Web 模拟微信咨询界面的后续实现。
- 负责 HTML、CSS、JavaScript 交互和基本可用性。
- 当前第 0 阶段不得实现聊天页面或业务交互。

## Knowledge Agent 职责
- 负责 Markdown 知识库结构、示例故障知识和检索素材。
- 确保知识内容不包含真实客户信息、账号、密码、密钥或真实内网 IP。
- 当前第 0 阶段只允许保留目录结构。

## Test Agent 职责
- 负责测试策略、测试用例和后续自动化测试。
- 覆盖后端接口、前端基础流程、mock Agent 输出和工单摘要。
- 当前第 0 阶段不得编写复杂测试代码。

## Docs Agent 职责
- 负责 README、项目简报、路线图、安全说明和多 Agent 工作流文档。
- 文档必须真实反映当前阶段，不得描述未完成能力为已完成。
- 负责记录阶段验收结果和后续建议。

## 允许修改的目录
- Main Agent：全仓库，但必须尊重阶段边界和最小变更原则。
- Backend Agent：`backend/`、`tests/` 中与后端相关内容、必要时更新 `docs/` 中后端说明。
- Frontend Agent：`frontend/`、`tests/` 中与前端相关内容、必要时更新 `docs/` 中前端说明。
- Knowledge Agent：`knowledge/`、`prompts/` 中知识与提示词相关内容、必要时更新 `docs/` 中知识库说明。
- Test Agent：`tests/`、必要的测试配置文件、必要时更新 `docs/` 中测试说明。
- Docs Agent：`README.md`、`AGENTS.md`、`docs/`、`prompts/` 中文档和提示词模板。

## 开发规则
- 每次任务必须先确认当前阶段目标和禁止事项。
- 每次变更应小步提交，优先完成可审查的最小闭环。
- 新功能必须先有清晰范围，再实现代码。
- 文档必须与实际代码状态一致。
- 后续接入真实微信、真实 LLM API 或外部服务前，必须先更新安全设计并通过 Main Agent 审查。
- 涉及用户输入、日志和工单摘要时，必须考虑隐私保护和敏感信息脱敏。

## 禁止事项
- 不允许一次性完成整个项目。
- 不允许提前实现下一阶段功能。
- 不允许在第 0 阶段实现 FastAPI 业务接口。
- 不允许在第 0 阶段实现前端聊天页面。
- 不允许接入真实微信、企业微信或微信客服接口。
- 不允许接入真实 LLM API。
- 不允许写入真实客户信息、账号、密码、密钥、真实 IP 或内部系统地址。
- 不允许引入复杂框架或与 MVP 无关的依赖。
- 不允许绕过 Main Agent 的阶段验收。

## 安全边界
- 第一版只使用本地 mock，不调用真实外部 AI 服务。
- 第一版只模拟微信类场景，不处理真实微信消息。
- 所有示例数据必须是假数据和通用问题。
- 不保存真实敏感信息。
- 日志和工单摘要不得包含密钥、密码、token、身份证号、手机号等敏感字段。
- 后续若需要真实集成，必须先设计鉴权、访问控制、数据留存、审计和脱敏机制。

## 每阶段验收要求
- 第 0 阶段：GitHub 仓库、基础目录、治理文档、多 Agent 工作流和路线图完成；不包含业务代码。
- 第 1 阶段：明确后端最小接口契约和 mock 数据流，通过基础测试；不接真实 LLM 或微信。
- 第 2 阶段：完成前端模拟咨询页面，与 mock 后端联通；不接真实微信。
- 第 3 阶段：完善知识库样例、问题分类和排查建议模板；仅使用假数据。
- 第 4 阶段：完成工单摘要生成、端到端演示和测试整理。
- 第 5 阶段：评估真实企业微信、微信客服或 LLM API 接入方案，但不得在未审查前直接接入。

## 阶段边界
当前阶段是第 0 阶段，只允许初始化仓库治理、目录结构和文档。任何业务代码、接口实现、页面实现、真实服务接入都必须留到后续阶段，并经过 Main Agent 明确授权。
