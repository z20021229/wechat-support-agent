# Version Plan

## 版本策略
项目采用小步版本演进。`v0.1.0` 冻结为初始 MVP，后续版本按能力边界推进，避免一次性引入真实微信、真实 LLM、数据库持久化和部署复杂度。

## v0.1.0：初始 MVP
目标：完成本地可演示的规则版技术支持 Agent。

范围：
- 类微信聊天页面。
- FastAPI 后端。
- 问题分类。
- Markdown 知识库。
- 知识库增强回复。
- 工单摘要生成器。
- GitHub 展示文档。

冻结说明：
- `v0.1.0` 作为初始阶段性成果提交。
- 当前不接真实微信。
- 当前不接真实 LLM API。
- 当前不实现 SQLite 和持久化。

## v0.2.0：多轮会话状态管理
目标：让 Agent 能追踪当前会话中已经收集的信息和仍缺失的信息。

计划：
- 多轮会话状态管理。
- 识别已收集信息。
- 识别缺失信息。
- `next_questions` 动态变化。
- `summary` 基于会话状态生成。

## v0.3.0：SQLite 持久化
目标：让会话和工单草稿可以保存、查询和导出。

计划：
- SQLite 保存会话。
- SQLite 保存工单。
- 查询历史工单。
- 导出工单。

## v0.4.0：知识库检索增强
目标：让回复能更精确引用 Markdown 知识库中的排查步骤。

计划：
- 知识库检索增强。
- 从 Markdown 中提取关键排查步骤。
- 回复引用知识库来源。

## v0.5.0：LLM Integration
目标：在保留 mock 模式的基础上接入真实 LLM API。

计划：
- 接入真实 LLM API。
- 支持配置 API Key。
- 支持 mock / real 两种模式。

安全前置：
- API Key 不写入仓库。
- 增加环境变量配置说明。
- 增加脱敏和日志边界。

## v0.6.0：WeChat Integration
目标：评估并设计企业微信或微信客服接入。

计划：
- 企业微信或微信客服接入方案。
- Webhook 设计。
- 消息收发流程。

安全前置：
- 明确鉴权方式。
- 明确消息验签方式。
- 明确数据留存边界。

## v1.0.0：Deployable Release
目标：形成可演示部署版本。

计划：
- Docker 部署。
- 安全配置。
- 日志审计。
- 可演示部署版本。
