# Development Workflow

## 主 Agent + 子 Agent 分工
本项目采用 Main Agent 监控 + 子 Agent 分工的方式开发。

- Main Agent：控制阶段边界、拆分任务、检查验收结果。
- Backend Agent：负责后端接口、分类、知识库读取和摘要生成。
- Frontend Agent：负责静态聊天页面。
- Knowledge Agent：负责 Markdown 知识库和支持话术模板。
- Docs Agent：负责 GitHub 展示文档和演示案例。

## 阶段开发内容
- 第 0 阶段：创建 GitHub 仓库、治理文件和目录结构。
- 第 1 阶段：创建 FastAPI mock API。
- 第 2 阶段：创建前端模拟微信聊天页面。
- 第 3 阶段：实现关键词问题分类模块。
- 第 4 阶段：创建 6 类 Markdown 技术支持知识库和 Prompt 文件。
- 第 5 阶段：让后端根据分类读取知识库并增强 `/chat` 回复。
- 第 6 阶段：实现规则版工单摘要生成器。
- 第 7 阶段：完善 GitHub 展示文档和演示案例。

## 每阶段验收机制
每个阶段都有明确的允许修改目录、禁止事项、测试命令和 commit message。完成阶段后执行验收，确认：
- 功能是否符合阶段目标。
- 是否没有越界修改。
- 是否没有接入真实微信或真实 LLM。
- 测试是否通过。
- git status 是否干净。
- 最新 commit 是否已 push。

## 为什么不让一个 Agent 一次性完成全部项目
一次性完成全部项目会让上下文过大，容易混入未来阶段功能，也容易忽略安全边界。分阶段开发可以保持每次变更可审查、可回滚、可验收。

## 如何避免 Prompt 过杂导致 Codex 降智
- 每阶段只给当前目标和禁止事项。
- 限制允许修改目录。
- 明确“不进入下一阶段”。
- 每阶段先实现最小闭环，再验收。
- 文档、后端、前端、知识库分角色推进，减少互相干扰。
